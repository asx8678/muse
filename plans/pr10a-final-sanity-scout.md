## 3. Findings

### Finding 1 ‚Äî HIGH ‚Äî Architecture module map overclaimed `Muse.resume`/`Muse.approve` public API

**File:** `docs/architecture.md` line 1099 (formerly 1095)
**Status:** ‚úÖ **FIXED by concurrent lane07** (commit `bc8eae9`)

**Previous stale text:**
```
lib/muse.ex                  Public API. Delegates submit/resume/approve to SessionServer.
```

**Current corrected text:**
```
lib/muse.ex                  Public API. Muse.submit/2 delegates to SessionRouter. Plan approval/rejection routed via /approve plan and /reject plan slash commands through SessionRouter ‚Üí SessionServer (not exposed as Muse.approve/‚Ä¶ public functions).
```

**Independent confirmation:** This scout independently identified the same module-map drift and recommended an identical replacement. The lane07 fix is correct and sufficient.

**Actual `Muse` module (`lib/muse.ex`):** Only `submit/2` exists. No `resume` or `approve` function.

**Reality:** Plan approval/rejection is routed through `/approve plan` and `/reject plan` slash commands ‚Üí `CommandDispatcher` ‚Üí `SessionRouter.approve_plan/2` ‚Üí `SessionServer.approve_plan/2`. There is no `Muse.approve/2` or `Muse.resume/2` public API function.

---

### Finding 2 ‚Äî MEDIUM ‚Äî SessionServer responsibilities shorthand `approve`/`reject` without `_plan` suffix

**File:** `docs/architecture.md` line 152
**Current text:**
```
- Synchronous calls: `submit`, `approve`, `reject`, `status`, `cancel`
```

**Actual `SessionServer` API:** The functions are `approve_plan/2` and `reject_plan/2`, not `approve`/`reject`.

**Recommended replacement:**
```
- Synchronous calls: `submit`, `approve_plan`, `reject_plan`, `status`, `cancel`
```

**Risk:** Low ‚Äî the context (SessionServer responsibilities) makes the shorthand mostly clear, but using the exact function names prevents ambiguity, especially since `approve`/`reject` could be confused with future patch or shell approval functions.

---

### Finding 3 ‚Äî LOW ‚Äî Testing checklist item about Coding Muse lacks roadmap note

**File:** `docs/testing.md` line 150
**Current text:**
```
- [ ] Coding Muse requires plan approval before write access is granted
```

**Context:** This unchecked checklist item reads as a current test to implement, but Coding Muse turn routing, `patch_propose`/`patch_apply` tools, and write access are all PR17+ roadmap scope. Other roadmap items in the same checklist area don't carry disclaimers either, so this is consistent ‚Äî but this specific item could mislead a reader into thinking Coding Muse write-access is near-term.

**Recommended replacement (if desired):**
```
- [ ] (PR17+) Coding Muse requires plan approval before write access is granted
```

**Risk:** Very low ‚Äî context makes it clear this is future scope. The "(PR17+)" prefix would make it unambiguous.
