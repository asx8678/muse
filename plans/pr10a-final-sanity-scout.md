## 3. Findings

### Finding 1 — HIGH — Architecture module map overclaims `Muse.resume`/`Muse.approve` public API

**File:** `docs/architecture.md` line 1095
**Current text:**
```
lib/muse.ex                  Public API. Delegates submit/resume/approve to SessionServer.
```

**Actual `Muse` module (`lib/muse.ex`):** Only `submit/2` exists. No `resume` or `approve` function.

**Reality:** Plan approval/rejection is routed through `/approve plan` and `/reject plan` slash commands → `CommandDispatcher` → `SessionRouter.approve_plan/2` → `SessionServer.approve_plan/2`. There is no `Muse.approve/2` or `Muse.resume/2` public API function.

**Recommended replacement:**
```
lib/muse.ex                  Public API. Muse.submit/2 delegates to SessionRouter. Plan approval/rejection routed via /approve plan and /reject plan slash commands through SessionRouter → SessionServer (not exposed as Muse.approve/resume public functions).
```

**Risk:** A reader building a client against the `Muse` module would look for `Muse.approve/2` and `Muse.resume/2` that don't exist. This is the most directly misleading finding.

---

### Finding 2 — MEDIUM — SessionServer responsibilities shorthand `approve`/`reject` without `_plan` suffix

**File:** `docs/architecture.md` line 152
**Current text:**
```
- Synchronous calls: `submit`, `approve`, `reject`, `status`, `cancel`
```

**Actual `SessionServer` API:** The functions are `approve_plan/2` and `reject_plan/2`, not `approve`/`reject`.

**Recommended replacement:**
```
- Synchronous calls: `submit`, `approve_plan`, `reject_plan`, `status`, `cancel`
```

**Risk:** Low — the context (SessionServer responsibilities) makes the shorthand mostly clear, but using the exact function names prevents ambiguity.

---

### Finding 3 — LOW — Testing checklist item about Coding Muse lacks roadmap note

**File:** `docs/testing.md` line 150
**Current text:**
```
- [ ] Coding Muse requires plan approval before write access is granted
```

**Context:** This unchecked checklist item reads as a current test to implement, but Coding Muse turn routing, `patch_propose`/`patch_apply` tools, and write access are all PR17+ roadmap scope. Other roadmap items in the same checklist area don't carry disclaimers either, so this is consistent — but this specific item could mislead a reader into thinking Coding Muse write-access is near-term.

**Recommended replacement (if desired):**
```
- [ ] (PR17+) Coding Muse requires plan approval before write access is granted
```

**Risk:** Very low — context makes it clear this is future scope. The "(PR17+)" prefix would make it unambiguous.
